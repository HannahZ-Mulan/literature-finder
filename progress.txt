## 优化详情 (2026-04-02)

### AI功能修复

#### 问题分析
用户反映AI中文解读和对话功能不工作，只给出浅薄的回复。经诊断发现：
1. **空内容问题**：部分论文（如论文17）的`extractedText`为空字符串
2. **错误状态标记**：这些论文仍被标记为`isComplete: true`
3. **AI收到空内容**：AI功能直接使用空的`extractedText`，导致泛泛而谈的回复
4. **缺乏验证**：上传和AI路由没有检查内容是否为空

#### 解决方案

**1. 修复上传API** (`src/app/api/papers/upload/route.ts:58-88`)
   - ✅ 添加文本长度验证（至少10个字符）
   - ✅ 提取失败时标记`isComplete: false`（而非true）
   - ✅ 记录详细的错误日志
   - ✅ 保留错误信息用于调试

**2. 添加AI聊天验证** (`src/app/api/papers/[id]/chat/route.ts:49-73`)
   - ✅ 检查`extractedText`是否为空或少于50字符
   - ✅ 返回清晰的错误消息和解决建议
   - ✅ 显示论文状态（ID、文件名、文本长度）

**3. 添加AI摘要验证** (`src/app/api/papers/[id]/summary/route.ts:67-73`)
   - ✅ 同样的空内容检查
   - ✅ 返回中文错误消息
   - ✅ 防止AI处理空内容

#### 测试结果

**论文20（有内容）：**
- ✅ AI对话正常工作
- ✅ DeepSeek提供详细的回答
- ✅ 回复基于实际论文内容（研究问题、方法等）

**论文17（空内容）：**
- ✅ 显示清晰的错误消息
- ✅ 告知用户可能的原因（扫描版、提取失败等）

## 优化详情 (2026-04-02) - PDF读取修复

### PDF读取问题修复完成

#### 问题分析
之前发现PDF上传后无法读取文本的问题，经诊断发现：
1. **数据库schema不匹配**：前端期待的字段（`totalPages`, `extractedPages`, `extractionMethod`）在schema中不存在
2. **UI过于复杂**：前端代码包含复杂的并行处理UI，但后端实际上使用简单的pdf-parse提取器
3. **架构不一致**：前端期待并行提取进度，后端只提供简单的完成状态

#### 解决方案

**1. 简化前端上传页面** (`src/app/upload/page.tsx`)
   - ✅ 移除 `PagePreview` 接口和 `pages` 状态
   - ✅ 简化 `Paper` 接口，移除不存在的字段
   - ✅ 简化轮询函数，只检查 `isComplete` 状态
   - ✅ 移除复杂的页面预览UI和进度百分比计算
   - ✅ 更新页面标题和描述，移除"并行提取"引用
   - ✅ 添加简单的提取状态显示

**2. 保持后端简单可靠**
   - ✅ 继续使用 `pdf-parse@1.1.1` (稳定版本)
   - ✅ 继续使用独立脚本 `scripts/extract-pdf-text.js`
   - ✅ 继续使用异步提取模式（不阻塞响应）
   - ✅ 数据库schema保持不变（简洁有效）

## 优化详情 (2026-03-09)

### AI解读和对话功能深度优化

#### 问题分析
上次测试发现AI解读文献和与AI对话功能存在以下问题：
1. 解读停留于表面，内容浅层且缺乏深度
2. 对话功能使用简单Mock模式，只有3个预设回答
3. 提示词缺少批判性思维和深度分析要求
4. 缺乏与现有工作的对比和创新点分析

#### 优化方案

**1. 增强解读提示词** (`src/lib/ai/prompts.ts`)
   - 添加专家级学术分析师角色定义
   - 增加创新点分析要求（理论创新、实践进展、与现有方法的区别）
   - 增加实验设计深度评估（具体算法名称、技术组件、理论基础）
   - 增加批判性评估要求（优势、局限性、替代视角）
   - 要求精确提取领域术语、具体指标和数字
   - 强调"不说空话"原则，要求具体而非泛泛而谈

**2. 重构对话系统提示词** (`src/lib/ai/openai.ts`, `src/lib/ai/deepseek.ts`)
   - 设计专业的学术研究助手角色
   - 六大核心指导原则：
     1. **精确具体**：引用论文原文，使用确切术语和数字
     2. **提供证据**：引用具体部分，解释作者选择理由
     3. **批判思维**：不限于总结，分析优缺点和替代观点
     4. **透明度**：明确说明哪些信息可用、哪些不可用
     5. **适当深度**：提供详细解释，使用例子和类比
     6. **学术严谨**：客观平衡，承认不确定性，避免夸大
   - 增加响应token限制（从1000增加到1500）

**3. 改进对话实现** (`src/app/api/papers/[id]/chat/route.ts`)
   - 移除简单的Mock模式（仅3个预设回答）
   - 集成AIManager进行真实AI对话
   - 添加智能降级方案：
     - 优先使用真实AI（OpenAI/DeepSeek）
     - AI不可用时提供有用的反馈信息
     - 引导用户配置API密钥以获得完整功能
   - 基于实际论文内容提供基本信息

#### 技术实现

**修改的文件：**
1. `src/lib/ai/prompts.ts` - 增强getExtractionPrompt提示词
2. `src/lib/ai/openai.ts` - 重构chatWithPaper系统提示词

4. `src/app/api/papers/[id]/chat/route.ts` - 集成真实AI功能，添加智能降级

### 修复的 Bug

#### 1. Drizzle Schema 属性名映射问题
**问题**：`SQLITE_CONSTRAINT_NOTNULL: NOT NULL constraint failed: uploaded_papers.file_name`

**原因**：
- Schema 使用 `file_name: text('file_name')`（snake_case）
- 代码插入时使用 `fileName`（camelCase）
- Drizzle 将属性名和列名视为同一个，导致映射错误

**修复**：
```typescript
// ❌ 之前（错误）
file_name: text('file_name').notNull(),
extracted_text: text('extracted_text').notNull(),

// ✅ 之后（正确）
fileName: text('file_name').notNull(),
extractedText: text('extracted_text').notNull(),
```

**影响文件**：
- `src/db/schema.ts` - 修改 uploadedPapers 表定义
- `src/app/api/papers/route.ts` - 修改查询使用 `papers.createdAt` 而非 `papers.created_at`

#### 2. 路由冲突问题
**问题**：`src/app/demo/route.ts` 与 `src/app/demo/page.tsx` 冲突

**错误**：
```
Error: You cannot have two parallel pages that resolve to the same path.
```

**修复**：删除 `src/app/demo/route.ts`（API 路由已存在于 `src/app/api/demo/route.ts`）

#### 3. OpenAI API 连接超时问题
**问题**：
```
ConnectTimeoutError: Connect Timeout Error (attempted address: api.openai.com:443)
```

**原因**：
- 在中国无法直接访问 `api.openai.com`
- `.env.local` 设置了 `AI_PROVIDER=mock` 但代码仍直接调用 OpenAI API

**修复**：
- 修改 `/api/papers/[id]/summary/route.ts` 使用 Mock 模式生成摘要
- 修改 `/api/papers/[id]/chat/route.ts` 使用 Mock 模式回答问题
- 生成的摘要和回答保存在数据库中
